# برنامه ترکیبی Achievementهای GitHub

آخرین بازبینی این سند: `2026-04-15`

این برنامه برای ریپوی `Auto` طراحی شده تا دو هدف را هم‌زمان نگه دارد:

1. متد فعلی اتوماسیون و ساخت contribution حفظ شود.
2. کیفیت بیرونی ریپو و فعالیت واقعی GitHub هم بالا برود.

## اصل مهم

فقط با زیاد کردن تعداد PR و Issue نمی‌توان همه Achievementها را با اطمینان گرفت.
بعضی Achievementها نیاز به رفتار واقعی در GitHub دارند و بعضی دیگر دیگر قابل کسب نیستند.

## وضعیت Achievementها

### قابل پیگیری در زمان نگارش این سند

- `Pull Shark`
- `Pair Extraordinaire`
- `Quickdraw`
- `YOLO`
- `Starstruck`
- `Galaxy Brain`
- `Public Sponsor`

### محدود یا خارج از کنترل مستقیم این ریپو

- `Public Sponsor`
  برای این مورد باید واقعا از طریق GitHub Sponsors حمایت مالی انجام شود.

- `Starstruck`
  این مورد به دریافت star از طرف دیگران وابسته است و با اتوماسیون داخلی به‌تنهایی حل نمی‌شود.

- `Galaxy Brain`
  این مورد به پاسخ‌های مفید و accepted answer در Discussions وابسته است؛ فقط با commit و PR به‌دست نمی‌آید.

### عملا غیرقابل کسب یا نامشخص

- `Arctic Code Vault Contributor`
  بازنشسته است و به snapshot تاریخی سال 2020 مربوط بوده است.

- `Heart On Your Sleeve`
- `Open Sourcerer`
  این دو badge در برخی فهرست‌ها دیده می‌شوند اما معیار عمومی و قابل اتکای فعالی برای کسب آن‌ها منتشر نشده است.

## استراتژی ترکیبی پیشنهادی

### لایه 1: حفظ متد فعلی

این ریپو می‌تواند همچنان برای این موارد به‌کار برود:

- ساخت contribution منظم
- ساخت branch و PR
- merge سریع برای `Quickdraw`
- افزایش تعداد PR merge شده برای `Pull Shark`
- co-authored commit برای `Pair Extraordinaire`

### لایه 2: افزودن فعالیت واقعی و باکیفیت

این لایه برای اعتبار پروفایل و badgeهایی که با اتوماسیون خالص به‌دست نمی‌آیند لازم است:

- ایجاد issue واقعی با template
- PRهای واقعی با توضیح مشخص
- مستندسازی بهتر
- test و workflow cleanup
- پاسخ‌دادن در GitHub Discussions
- جذب star از طریق README بهتر، release و معرفی پروژه

## نقشه عملی برای هر Achievement

### Pull Shark

- متد فعلی را نگه دارید.
- به‌جای تکیه کامل بر حجم بالا، بخشی از PRها را واقعی‌تر و موضوع‌دارتر کنید.
- چند PR در هفته با عنوان و توضیح بهتر بسازید.

### Pair Extraordinaire

- بخشی از commitها را به‌صورت co-authored انجام دهید.
- برای هر contribution campaign حداقل 1 یا 2 commit مشترک واقعی داشته باشید.

### Quickdraw

- فقط برای تعداد کمی PR از merge سریع استفاده کنید.
- بهتر است Quickdraw روی PRهای مشخص و کم‌ریسک انجام شود، نه روی همه چیز.

### YOLO

- یک بار کافی است.
- آن را روی یک PR بسیار کوچک، واضح و کم‌ریسک انجام دهید.

### Starstruck

- README را محصول‌محورتر کنید.
- release/tag منظم داشته باشید.
- مثال خروجی واقعی، quickstart کوتاه و مزیت پروژه را روشن کنید.
- پروژه را در شبکه‌های اجتماعی و بین دوستان توسعه‌دهنده معرفی کنید.

### Galaxy Brain

- در GitHub Discussions ریپوهای دیگر پاسخ دقیق و مفید بدهید.
- روی Discussionهایی تمرکز کنید که احتمال accepted answer شدن دارند.
- این badge را نباید به این ریپو محدود کرد.

### Public Sponsor

- اگر واقعا هدفتان همه badgeهای قابل کسب است، این مورد فقط با sponsor واقعی به‌دست می‌آید.

## cadence پیشنهادی

### روزانه

- 1 contribution خودکار
- حداکثر 1 PR سریع برای هدف‌های count-based

### هفتگی

- 1 issue واقعی
- 1 PR واقعی برای بهبود پروژه
- 1 cleanup یا test task
- 1 discussion answer بیرون از این ریپو

### ماهانه

- 1 release
- 1 بازبینی README
- 1 کمپین برای گرفتن star و بازخورد

## چیزهایی که بهتر است کنترل شوند

- تعداد PRهای انبوه روزانه
- reviewهای خودکار با متن بیش از حد کلی
- issueهای مصنوعی که بلافاصله بسته می‌شوند
- ادعاهای مستنداتی که با رفتار واقعی workflowها نمی‌خوانند

## تعریف موفقیت

موفقیت در این برنامه یعنی:

- متد فعلی حفظ شود
- نمودار contribution فعال بماند
- سیگنال‌های واقعی همکاری بیشتر شوند
- شانس badgeهای قابل کسب بالا برود
- ظاهر ریپو برای بازدیدکننده حرفه‌ای‌تر شود
